## Evidence

Implemented a full modular backend CRUD layer using:

```
router -> service -> SQLAlchemy ORM model
```

Main implementation details:

- Added entity-specific FastAPI routers for all core entities, organized into domain subfolders under `routers/v1/`.
- Added entity-specific service modules that encapsulate all database operations per entity.
- Added centralized router registration through a single `v1_router` under `/api/v1`.
- Used `DbSession = Annotated[Session, Depends(get_db)]` alias for cleaner FastAPI dependency injection.
- Added router-level tests for all CRUD endpoints using pytest + FastAPI TestClient with an in-memory SQLite database for full test isolation.
- Added user-specific service logic for password handling:
  - API accepts plain `password` on user create/update.
  - `user_service` hashes the password before storing it as `password_hash`.
  - User API responses do not expose `password` or `password_hash`.
- Each router has its own Swagger tag so endpoints are grouped by model in `/docs`.

## Notes for the mentor

This PR introduces the full CRUD API layer for the current core schema. Routers are organized into domain subfolders (`farms`, `crops`, `sensors`, `harvests`, `reference_data`, `users`) under `routers/v1/`, with a single `v1_router` registered in `main.py`.

Each entity has its own service module that keeps database logic separate from HTTP handling in the router. Entity-specific services are intentionally thin for now but provide clear extension points for future custom queries, validation rules, and business logic.

The main exception is `user_service`, where create and update operations include password hashing logic. This keeps password handling out of the router layer and prevents exposing sensitive fields through API responses.

Tests use an in-memory SQLite database with `StaticPool` for connection sharing across requests, and a `BigInteger->Integer` type patch to work around SQLite's autoincrement behaviour.
